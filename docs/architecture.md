# Architecture

> Главный принцип: **ИИ — для творческих и интерпретационных задач, код — для всего остального.** Каждый ИИ-вызов медленный, дорогой и недетерминированный. Минимизируй, кешируй, шаблонизируй.

## Работа с фандомами — без ручной курации

Знания фандомов уже зашиты в LLM на уровне претрейна. Не курируем вручную, не пишем wiki, не сидим выписывать каноны. Так работают character.ai, NovelAI, AI Dungeon — так работаем и мы.

- Юзер вводит фандом + персонажей текстом, плюс автокомплит по самым частым запросам в базе
- LLM сам строит character bible при генерации первой главы
- Character reference картинка генерится **один раз** на персонажа, кешируется по `(fandom_normalized, character_name_normalized)`, переиспользуется во всех сценах и историях
- Опциональный community layer: если N юзеров сгенерили одного персонажа, его reference и bible можно шарить как «community canon»
- Custom миры (оригинальный фандом юзера) — тот же flow, в коде нет разницы между Геншином и «миром, который Маша придумала»

## Высокоуровневый pipeline (глава → видео-эпизод)

1. Юзер вводит промпт (фандом + персонажи + тропы + premise)
2. **[ИИ]** Генерация главы + character bible
3. **[ИИ]** Парсинг главы на сцены — структурированный JSON
4. **[Код]** Сборка промпта для картинки из шаблона
5. **[Кеш + ИИ]** Генерация картинок (Flux + character reference)
6. **[Код]** Валидация (face detection, проверка качества)
7. **[ИИ, опционально]** Image-to-video через Kling / Runway
8. **[ИИ]** TTS-озвучка через ElevenLabs (voice_id из bible)
9. **[Код]** FFmpeg склейка финального эпизода

## ИИ vs код по шагам

| Шаг                             | ИИ или код                          | Почему                                                    |
| ------------------------------- | ----------------------------------- | --------------------------------------------------------- |
| Создание character bible        | ИИ один раз                         | Творческая интерпретация                                  |
| Хранение/выдача персонажей      | Код                                 | CRUD                                                      |
| Разбиение текста на сцены       | ИИ                                  | Семантика                                                 |
| Определение кто в сцене         | Гибрид: regex + ИИ для местоимений  | Большинство случаев решается кодом                        |
| Выбор камеры (closeup/wide/etc) | **Код по правилам**                 | Design pattern, не творчество. 10–15 правил покрывают 90% |
| Эмоция сцены                    | ИИ → enum из 8–12 значений          | Кешируется                                                |
| Сборка промпта для картинки     | **Код, шаблон**                     | Из структурированного JSON — детерминированный промпт     |
| Генерация картинки              | ИИ                                  | Очевидно                                                  |
| Валидация результата            | Код (face detection)                | Программная проверка                                      |
| Motion prompt для видео         | **Код, шаблон по типу сцены**       | Десяток шаблонов покрывают 90%                            |
| Музыка                          | **Код, лицензированная библиотека** | Не ИИ. Дешевле, быстрее, стабильнее                       |
| TTS озвучка                     | ИИ                                  | Очевидно                                                  |
| Финальная склейка видео         | Код, FFmpeg                         | Просто склейка                                            |

## Архитектурные правила

1. **Структурированный JSON-выход у каждого ИИ-вызова.** Strict schema. Никогда не парсить свободный текст regex-ом.
2. **Семантический кеш.** Хешировать структурированный вход (`character_ids + scene_type + emotion + location_id`), не сырой промпт. Близкие сцены попадают в кеш.
3. **Версионирование промптов.** Сохранять `template_id + version` с каждым ассетом. Улучшила промпт — старые ассеты можно перегенерить партией.
4. **Cost tracking.** Логировать стоимость каждого ИИ-вызова. Метрика: «средний эпизод $X.XX, parsing $0.02, images $1.50, videos $25, voice $0.30».
5. **Job queue для всего >5 секунд.** Парсинг — sync. Картинки — async. Видео — async с push-уведомлением.
6. **Идемпотентность каждого шага.** Упало на видео-генерации — перезапуск только этого шага, не всего пайплайна с нуля.
7. **Пресгенерированные ассеты.** Для каждого фандома один раз генеришь ~50 фоновых локаций (библиотека, гостиная, лес). Дальше переиспользуются между эпизодами через inpainting. Стоимость кадра падает в 3–5 раз.

## Data model — первый набросок

| Таблица                                 | Назначение                                                                                |
| --------------------------------------- | ----------------------------------------------------------------------------------------- |
| `users`                                 | Профили (Supabase Auth)                                                                   |
| `stories`                               | Истории (title, fandom, premise, owner_id, visibility)                                    |
| `chapters`                              | Главы (story_id, ordinal, text, status)                                                   |
| `characters`                            | Персонажи в bible (story_id, name, description, voice_id)                                 |
| `character_references`                  | Кеш картинок-референсов по `(fandom_normalized, character_name_normalized)`               |
| `scenes`                                | JSON-структура сцен (chapter_id, scene_index, emotion, who_in_scene, camera, location_id) |
| `assets`                                | Сгенерённые картинки/видео (scene_id, type, url, cost, template_id, template_version)     |
| `jobs`                                  | Очередь генераций (type, status, payload, retries)                                        |
| `likes`, `saves`, `comments`, `follows` | Социалка                                                                                  |

Финальная Prisma-схема пишется на День 1 Phase 1.

## Реалистичные ожидания по визуалам

| Сцена                     | Узнаваемость персонажа |
| ------------------------- | ---------------------- |
| 1 персонаж крупным планом | ~85%                   |
| Двое в кадре              | 65–75%                 |
| Трое+ персонажей          | падает быстро          |
| Полноростовой кадр        | хуже close-up          |
| Действие (драка, бег)     | хуже статики           |

**Продуктовое следствие:**

- Основные сцены — крупные/средние планы с 1–2 героями
- Avoid полноростовых кадров
- Показывать 2–4 варианта одной сцены, дать юзеру выбрать
- Кнопка «перегенерировать» в один клик — обязательна
- Wide-shots с массовкой — пейзажные, без узнаваемых лиц на заднем плане

## Test learnings

**Kling Omni переписывает фон под промпт.** Если в motion prompt описать освещение/атмосферу — модель перерисует сцену. Для preserve-режима в шаблон motion_prompt **НЕ включать** описание света и фона. Только движение и камера.

```javascript
// motion_prompt template
{character_action}. {camera_movement}. {micro_motion}.
```

Для сохранения исходной локации между сценами одной главы — использовать обычный Image-to-Video режим, **не Omni**.
